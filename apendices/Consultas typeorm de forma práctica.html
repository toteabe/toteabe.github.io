Entity

La tabla de departamento se respalda mediante la entidad:

import {Column, Entity, OneToMany, PrimaryGeneratedColumn} from "typeorm";  
import {Empleado} from "../empleados/empleado.entity";  
  
@Entity()  
export class Departamento {  
  
    @PrimaryGeneratedColumn()  
    id!: number;  
  
    @Column("varchar", {length: 255, unique: true} )  
    nombre!:string  
  
    @Column({ type: 'numeric', precision: 12, scale: 2 })  
    presupuesto!: string;  
  
    @Column({ type: 'numeric', precision: 12, scale: 2 })  
    gasto!: string;  
  
    @OneToMany(()=> Empleado, (e) => e.departamento)  
    empleados!:Empleado[];  
  
}

Los decoradores utilizados son:

  • @Entity(), marca la clase para uso por el gestor de entidades de typeorm, entityManager.
  • @PrimaryGeneratedColumn(), indica campo de clave primaria autogenerada.
  • @Column(“varchar”, {length: 255, unique: true} ), define el tipo utilizado para la columna en la tabla, varchar, junto con propiedades de longitud y unicidad. Si quieres que se sincronice -se cree en bbdd- la propiedad tienes que establecer con este decorador.
  • @OneToMany(()=> Empleado, (e) => e.departamento) , establece la relación uno-a-muchos de departamento a empleado sobre el array de empleados.

Por el lado de muchos se tiene a la tabla empleado:

import {Column, Entity, Index, JoinColumn, ManyToOne, PrimaryGeneratedColumn} from "typeorm";  
import {Departamento} from "../departamentos/departamento.entity";  
  
@Entity()  
export class Empleado{  
  
    @PrimaryGeneratedColumn()  
    id!: number;  
  
    @Column({type: "varchar",  length: 9, nullable: false, unique:true})  
    nif!: string;  
  
    @Column({type: "varchar",  length: 100, nullable: false})  
    nombre!: string;  
  
    @Column({type: "varchar",  length: 100, nullable: false})  
    apellido1!: string;  
  
    @Column({type: "varchar",  length: 100, nullable: true})  
    apellido2?: string | null;  
  
    @Index()  
    @Column({name: "departamento_id", type: 'int', nullable: true, default: null})  
    departamentoId?: number | null;  
  
    @ManyToOne(type => Departamento,  
        (d) => d.empleados,  
        { nullable: true, onDelete: "CASCADE" })  
    @JoinColumn({ name: "departamento_id" })  
    departamento!: Departamento;  
  
}

De esta entidad hay que destacar el atributo de departamentoId:

    @Index()  
    @Column({name: "departamento_id", type: 'int', nullable: true, default: null})  
    departamentoId?: number | null;

Este campo mapea la clave foránea (FK) de la tabla departamento en empleado. Al ser una FK de un modelo Typeorm debe haber un índice, en un esquema con la propiedad de synchronize: true se traduce en la creación del índice.

create index "IDX_bc7723b9a48814a4571aa92edc"  
    on empleado (departamento_id);

Por otro lado, se tiene la entidad hacia la parte uno, departamento:

    @ManyToOne(type => Departamento,  
        (d) => d.empleados,  
        { nullable: true, onDelete: "CASCADE" })  
    @JoinColumn({ name: "departamento_id" })  
    departamento!: Departamento;

Un aspecto importante es el nombre de departamento_id de la columna de FK sobre el decorador @JoinColumn, idéntico al de la propiedad departamentoId.

En este caso, no se trata de una duplicidad, sino una técnica para hacer explícito el id de FK en la entidad hija.

En Typeorm, ni EAGER ni LAZY, por defecto no se cargan las relaciones salvo indicación.

La configuración por defecto para cualquier relación (incluyendo OneToMany) no es ni EAGER ni LAZY de forma automática; por defecto, las relaciones no se van cargar a menos que se especifique explícitamente en la consulta.

DataMapper Repository

En el snippet siguiente se utilizan 2 repositorios de Typeorm sobre patrón DataMapper:

  • empleadoRepo - objeto repository para la entidad Empleado.

En primer lugar se busca una entidad de Departamento mediante el método pre-implementado en el repository findOne.


let departamento  = await departamentoRepo.findOne({where: {nombre: "Informática"}});

El objeto que recibe como argumento es un FindOption. # FindOptions FindOptions son los objetos que puedes pasar a los métodos .find* de un Repository o EntityManager (find, findOne, findBy, findAndCount, etc.) para construir consultas sin usar QueryBuilder. Find options

Se dividen en: ## Opciones básicas

  • select: qué columnas del entity quieres devolver.
userRepository.find({
  select: {
    firstName: true,
    lastName: true,
  },
})
// SELECT "firstName", "lastName" FROM "user"
  • relations: qué relaciones cargar (incluye sub‑relaciones).
userRepository.find({
  relations: {
    profile: true,
    photos: true,
    videos: { videoAttributes: true },
  },
})
// ... FROM "user" 
// LEFT JOIN "profile" ON "profile"."id" = "user"."id" 
// LEFT JOIN "photo" ON "photo"."user_id" = "user"."id"
// LEFT JOIN "video" ON "video"."user_id" = "user"."id"
// LEFT JOIN "video_attribute" ON "video_attribute"."video_id" = "video"."id"...
  • where: condiciones simples (AND por defecto, o un array para OR).
userRepository.find({
  where: {
    firstName: "Timber",
    lastName: "Saw",
  },
})
// WHERE "firstName" = 'Timber' AND "lastName" = 'Saw'

userRepository.find({
  where: [
    { firstName: "Timber", lastName: "Saw" },
    { firstName: "Stan", lastName: "Lee" },
  ],
})
// (...Timber AND Saw) OR (...Stan AND Lee)

También puedes filtrar por propiedades de relaciones:

userRepository.find({
  relations: { project: true },
  where: {
    project: { name: "TypeORM", initials: "TORM" },
  },
})

// ... FROM "user"
// LEFT JOIN "user"."project_id" ON "project"."id"
// WHERE "project"."name" = 'TypeORM' AND "project"."initials" = 'TORM'
  • order: orden de los resultados.
userRepository.find({
  order: {
    name: "ASC",
    id: "DESC",
  },
})

// ... ORDER BY "user"."name" ASC, "user"."id" DESC
  • withDeleted: incluir registros soft‑deleted.
userRepository.find({
  withDeleted: true,
})
/* Obtiene incluso entidades con soft-delete:

@Entity() 
export class User { 
    @PrimaryGeneratedColumn() 
    id: number 
    
    @Column() 
    name: string 
    
    
    @DeleteDateColumn()
    //Propiedad que implementa el soft-delete, mientras sea NULL no 
    //se ha realizado el soft-delete
    deletedAt: Date 
}
*/
  • skip y take: paginación (offset/limit).
userRepository.find({
  order: { columnName: "ASC" },
  skip: 0,
  take: 10,
})
  • cache y lock: cacheo de resultados y modos de bloqueo (solo en findOne/findOneBy).
const users = await userRepository.find({
  where: { isAdmin: true },
  cache: {
    id: "users_admins",  //identificador de caché para invalidar después
    milliseconds: 25000, //tiempo de vigencia del valor en caché 25s
  },
})

...
await dataSource.queryResultCache.remove(["users_admins"])

Operadores avanzados en where:

En where puedes usar operadores para comparaciones más complejas, por ejemplo:

import { LessThan, MoreThan, In, Like, Between, IsNull } from "typeorm"

dataSource.getRepository(Post).findBy({
  likes: Between(1, 10),
  title: Like("%out #%"),
  authorId: In([1, 2, 3]),
  deletedAt: IsNull(),
})

También existen combinadores como And, Or, Not:

import { Or, Equal, ILike } from "typeorm"

const posts = await dataSource.getRepository(Post).findBy({
  title: Or(Equal("About #2"), ILike("About%")),
})

// ILIKE case-insensitive LIKE en postgres

Y Raw para expresiones SQL personalizadas:

import { Raw } from "typeorm"

const posts = await dataSource.getRepository(Post).findBy({
  currentDate: Raw((alias) => `${alias} > :date`, { date: "2020-10-06" }),
})

//SELECT * FROM "post" WHERE "currentDate" > '2020-10-06'

En resumen, FindOptions es el objeto que combina estas propiedades (select, relations, where, order, skip, take, etc.) para definir qué y cómo quieres leer datos con los métodos .find* sin escribir SQL ni usar QueryBuilder directamente.

repository.create() se usa para crear instancias de entidad en memoria (objetos TypeORM) a partir de nada o de un objeto plano, sin guardarlas todavía en la base de datos.

Este método sólo crea la entidad; no hace ningún INSERT ni UPDATE.

Es útil para:

  • Mapear datos de un DTO / body de request a una entidad.

  • Asegurarte de que se aplican los tipos y valores por defecto definidos en la entidad.

  • Para persistir lo que creaste con create, debes usar luego  repository.save(entity).

const creado = await empleadoRepo.save(empleado);  
  
//Jest
expect(creado.id).toBeDefined();  
expect(creado.departamentoId).toBe(departamento.id);

Cómo funciona el método save:

save

  • Inserta o actualiza según si la entidad ya existe (por su primary key).

  • Acepta entidades completas o parciales.

  • Usa una transacción para todas las entidades dadas.

  • Omite propiedades undefined (útil para actualizaciones parciales).

  • Devuelve la(s) entidad(es) guardada(s).

await repository.save(user) await repository.save([user1, user2])

Alternativas a save:

insert

  • Solo inserta filas nuevas.

  • No intenta buscar ni actualizar registros existentes.

  • Acepta objetos literales o arrays.

  • Es útil para inserciones masivas y simples.

await repository.insert({ firstName: "Timber", lastName: "Timber" }) 

await repository.insert([{ firstName: "Foo" }, { firstName: "Bar" }])

upsert

  • Inserta o actualiza en una sola operación SQL, según conflicto en columnas que tú defines.

  • Requiere indicar conflictPaths (o array simple en la forma corta) para saber cuándo hay conflicto.

  • Soportado en AuroraDataApi, Cockroach, MySQL, Postgres y SQLite.

  • Cuando hay actualización por conflicto, actualiza automáticamente columnas especiales como @UpdateDateColumn y @VersionColumn.

Forma corta:

await repository.upsert( [ { externalId: "abc123", firstName: "Rizzrak" }, {
 externalId: "bca321", firstName: "Karzzir" }, ], ["externalId"], )

Forma con más opciones:

await repository.upsert( [ { externalId: "abc123", firstName: "Rizzrak" }, 
{ externalId: "bca321", firstName: "Karzzir" }, ], { conflictPaths: 
["externalId"], skipUpdateIfNoValuesChanged: true, upsertType: "upsert",  //según driver 
}, )

QueryBuilder

En TypeORM puedes crear un QueryBuilder directamente desde un repositorio para construir consultas SQL complejas sobre la entidad de ese repositorio.

Ejemplo básico:

const users = await dataSource .getRepository(User) .createQueryBuilder("user") // "user" es el alias de la tabla
.where("user.name = :name", { name: "John" }) 
.getMany()

//SELECT * FROM "user"
//WHERE "user"."name" = :name

Aquí:

  • getRepository(User) obtiene el repositorio de la entidad User.

  • createQueryBuilder("user") crea un SelectQueryBuilder con alias "user".

  • Luego encadenas métodos (where, getMany, etc.) para construir y ejecutar la consulta.

Ejemplo con joins y paginación:

const users = await dataSource .getRepository(User) .createQueryBuilder("user") 
.leftJoinAndSelect("user.photos", "photo") 
.where("user.isActive = :active", { active: true }) 
.orderBy("user.id", "DESC") 
.skip(5) // offset 
.take(10) // limit 
.getMany()

//SELECT "user".*, "photo".* FROM "user" 
//LEFT JOIN "photo" ON "photo"."user_id" =  "user"."id"
//WHERE "user"."is_active

  • leftJoinAndSelect carga relaciones (similar a relations en find).

  • skip y take implementan paginación.

Cuándo usar QueryBuilder en lugar de find

  • Cuando necesitas joins complejos, condiciones avanzadas, subconsultas, ordenamientos y paginación en una sola consulta.

  • find/findOne con opciones (where, relations, order, skip, take) cubren casos simples; QueryBuilder es para consultas más elaboradas.

Devuelve una lista con el nombre de los departamentos que no tienen un presupuesto entre 100000 y 200000 euros. Sin utilizar el operador BETWEEN.

const presuInf = "100000";  
const presuSup = "200000";  
  
const lista = (  
                        await departamentoRepo  
                            .find({select: ["nombre"],  
                                    where: { presupuesto: Not(And(MoreThanOrEqual(presuInf)  
                                                            , LessThanOrEqual(presuSup)))  
                                            }  
                                    })  
                    ).map(d => d.nombre)

Explicación:

  1. select: ["nombre"]
    Solo se selecciona la columna nombre de la entidad (selección parcial con find).

  2. where: { presupuesto: ... }
    Se filtra por la columna/propiedad presupuesto usando operadores.

  3. MoreThanOrEqual(presuInf)
    Genera la condición presupuesto >= presuInf.

  4. LessThanOrEqual(presuSup)
    Genera la condición presupuesto <= presuSup.

  5. And(MoreThanOrEqual(...), LessThanOrEqual(...))
    Combina ambas condiciones con AND, es decir:
    presupuesto >= presuInf AND presupuesto <= presuSup.

  6. Not(And(...))
    Niega todo el rango, quedando:
    NOT (presupuesto >= presuInf AND presupuesto <= presuSup)
    Es decir, devuelve los departamentos cuyo presupuesto está fuera del intervalo [presuInf, presuSup].

  7. .map(d => d.nombre)
    De la lista de entidades devueltas, te quedas solo con el array de nombres.

Devuelve una lista con el nombre de los departamentos, gastos y presupuesto, de aquellos departamentos donde los gastos sean mayores que el presupuesto del que disponen.


const lista = await datasource.sql`SELECT D.* from departamento D where D.gasto > D.presupuesto`

Explicación:

  • datasource.sql\…`: usa el *tag* sqldelDataSource, que es un wrapper sobre .query`, pero con sintaxis de template literal. [SQL tag]

  • Dentro del template literal escribes SQL “normal”. En este caso:

    • SELECT D.* FROM departamento D: selecciona todas las columnas (*) de la tabla departamento, con alias D.

    • WHERE D.gasto > D.presupuesto: filtra solo los departamentos cuyo gasto es mayor que su presupuesto.

Como no hay expresiones ${...} dentro del template, no se están usando parámetros; es simplemente una consulta SQL directa ejecutada a través de TypeORM. El resultado (lista) será un array de filas “raw” (objetos planos) devueltos por la base de datos, igual que con dataSource.query(...).

También se tiene la variante:


const lista = await datasource.query("SELECT D.* from departamento D where D.gasto < D.presupuesto");

Explicación:

  • dataSource.query(sql, params?) ejecuta una consulta SQL tal cual en la base de datos y devuelve los resultados como filas “raw” (objetos planos), sin mapearlos a entidades.

const dept = (  
                await departamentoRepo.createQueryBuilder("D")  
                .select("D.nombre")  
                .distinct()  
                .innerJoin("D.empleados", "E")  
                .where("E.nif = '38382980M'")

Explicación:

  1. departamentoRepo.createQueryBuilder("D")
    Crea un SelectQueryBuilder para la entidad del repositorio, usando "D" como alias. Es equivalente a select("D").from(Departamento, "D").

  2. .select("D.nombre")
    Selección parcial: solo se selecciona la columna nombre del alias D, no todas las columnas de Departamento. Es el mismo patrón que select(["user.id", "user.name"]).

  3. .distinct()
    Añade DISTINCT al SELECT, de forma que si hubiera varios empleados con ese NIF asociados al mismo departamento, solo se devuelva un valor único de D.nombre.

  4. .innerJoin("D.empleados", "E")
    Hace un INNER JOIN con la relación empleados de Departamento, usando "E" como alias. Es análogo a los ejemplos de innerJoin("user.photos", "photo").

  5. .where("E.nif = '38382980M'")
    Añade la condición WHERE sobre la tabla/alias E (empleados). Es el mismo patrón que where("user.name = :name", { name: "Timber" }), solo que aquí se ha puesto el valor literal en la cadena.

    (En un código real se recomienda usar parámetros: where("E.nif = :nif", { nif: "38382980M" }) para evitar inyección SQL).

  6. .getRawOne()
    Ejecuta la consulta y devuelve un solo resultado raw (no una entidad). Para selecciones específicas o agregadas se usa getRawOne/getRawMany.

  7. ).D_nombre;
    Cuando haces una selección parcial con alias de tabla, TypeORM genera nombres de columnas en el resultado raw con el patrón <alias>_<propiedad>.
